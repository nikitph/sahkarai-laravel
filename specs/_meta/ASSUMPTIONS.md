# Assumptions

These behaviours are **inferred** from the PRD context but not literally stated. Each one is
flagged so Product can confirm or correct before the affected scenarios are taken as authoritative.

| ID | Assumption | Why Needed | Impact |
|----|------------|------------|--------|
| ASSUMPTION-A001 | Generic sign-in error reason is `invalid_credentials` (does not disclose whether the email exists) | Required for the "wrong password" and "unknown email" scenarios to assert the same response shape | Without this the platform may leak account-existence information on the sign-in surface |
| ASSUMPTION-A002 | Password-reset response is the same string regardless of whether the email is registered | Standard non-enumeration practice for password-reset; not stated in PRD | Without this, the reset endpoint may leak which emails are registered |
| ASSUMPTION-A003 | Password-reset token TTL is 60 minutes (Supabase Auth provider default) | The §4.5 invalid-token boundary scenario requires a concrete TTL | Wrong TTL value makes the boundary test wrong |
| ASSUMPTION-A004 | Pagination default page size is 20 for the Archive browse surface | Browse scenarios require concrete page-size boundaries | A different default makes the boundary examples wrong (sizes can be re-parameterised easily) |
| ASSUMPTION-A005 | A signed-in user with role `individual_member` is the only role that can submit Issue Reports | PRD §5.5 lists capture fields including user_id but does not explicitly forbid saas_admin reports; assumed staff use admin notes instead | Wrong scoping would change rbac scenarios in `interpretation_report_issue.feature` |
| ASSUMPTION-A006 | Razorpay subscription state changes are the source of truth and reconcile-job drift triggers an ops alert (not auto-fix) | PRD §10.6 says "detects drift" without specifying remediation | If automated remediation is desired, the reconcile scenario needs to be expanded |
| ASSUMPTION-A007 | Empty applicability_tags array is valid (`applicability_tags=[]`) | §5.2 says "zero or more"; we assume empty is acceptable | If empty is invalid, payload validation must reject it |
| ASSUMPTION-A008 | A user-initiated cancellation of a queued downgrade is possible before its effective date | §10.4 says downgrades are queued; PRD does not explicitly allow or forbid rescinding | If rescinding is forbidden, the cancellation scenario in `payment_downgrade.feature` must be removed |
| ASSUMPTION-A009 | Per-Source notification preferences are auto-seeded at signup with `enabled=true, email_cadence=daily_digest` for {RBI, IT, GST} | §7.4 says "default enabled at signup for all three" and the default cadence is daily_digest; the seeding moment is not explicit | If preferences are not auto-seeded, the default-state scenario must read "absent preference is treated as enabled+daily_digest" instead |
| ASSUMPTION-A010 | "Chat is bound to one Document Version at creation" forbids re-binding entirely (not just at creation) | §8.1 phrasing is "bound at creation and cannot be re-bound"; we encode this as a permanent immutability | None — same intent restated |
| ASSUMPTION-A011 | The post-soft-delete restore flow uses a distinct surface from the standard sign-in (e.g. an alternative restore action) | §3.4 plus §9.2 imply standard sign-in won't work while soft-deleted; restore must therefore have its own affordance | If restore happens through standard sign-in, `signin.feature`'s "account_pending_deletion" scenario should instead invoke restoration |
| ASSUMPTION-A012 | The "Versions" panel is shown only when more than one version exists | §6.3 says "for revisions" — we assume single-version Documents do not render the panel | Wrong = panel always renders with one entry |
| ASSUMPTION-A013 | A `saas_admin` cannot read another user's full credit ledger detail (only aggregated metadata) | §2.5 lists aggregated read access; full ledger detail is not enumerated as readable | If staff need full ledger detail for billing-support cases, RLS must allow it |
| ASSUMPTION-A014 | Razorpay event names used in webhooks are the standard Razorpay Subscriptions API event names (subscription.activated, etc.) | §10.6 names webhooks generally; specific event identifiers are not enumerated | Names can be re-parameterised in `payment_webhooks.feature` |
| ASSUMPTION-A015 | The configurable context-window threshold is read from configuration at request time, not baked into a build constant | §14 item 3 calls this out as an open placeholder | If hard-coded, threshold changes require redeploy and `chat_context_full.feature` must be parameterised differently |
| ASSUMPTION-A016 | A backfilled Document Version is identified by which job created it (the one-off batch job) and is the basis of the no-Notification rule | §4.7 says backfilled documents don't trigger notifications, without naming the discriminator | If backfill status is tracked differently (e.g. ingested_at < a launch cutoff), the discriminator differs |
| ASSUMPTION-A017 | Free users receive no notifications at all (not even for revisions they viewed) | §2.3 matrix denies "Receive Notifications" for Free; §7.2 trigger phrasing might suggest Free users who viewed a doc still get revision notifications | The matrix wins (§2.3) — but flag this if Product intended otherwise |
| ASSUMPTION-A018 | "Adjustment" credit_ledger entries are writeable only by `saas_admin` via an authenticated admin route or by service-role | §12.3 lists `adjustment` as a ledger reason without specifying who writes it | Wrong actor would change rbac scenarios |
| ASSUMPTION-A019 | An expired access token can be silently refreshed by the client when the refresh token is valid (Supabase default behaviour) | §3.4 says access tokens rotate per Supabase defaults | None — provider-default behaviour |
