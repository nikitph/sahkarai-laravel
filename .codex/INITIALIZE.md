# Company SaaS foundation initialization

You are working inside a newly generated **official Laravel React starter**. Transform it
into the company SaaS foundation described below. Implement the work completely and verify
it; do not stop after producing a plan.

The current upstream application is authoritative for Laravel framework and authentication
structure. Preserve its current versions and conventions unless a requirement below needs a
targeted change. Do not replace it wholesale with files from an older starter or reference app.

Read `.codex/ACCEPTANCE.md` before editing. Treat it as the definition of done.

## Product input

Before implementation, determine from the repository name or ask only if genuinely unknown:

- Product display name
- Composer package name (`company/<product-slug>` by default)
- Local HTTP port if the default is occupied

Do not wait for branding assets. Use a tasteful typographic/product-mark placeholder that is
easy to replace.

## Locked technical decisions

- Current Laravel major generated by the initializer, PHP 8.4
- Inertia, React, TypeScript, Tailwind CSS, and shadcn/ui conventions
- Client-side rendering; do not introduce Inertia SSR
- PostgreSQL in production and Docker development
- Redis for cache, queues, and broadcasting support
- Laravel Reverb for WebSockets
- Laravel queues and scheduler as independent long-running service roles
- FrankenPHP runtime image `ghcr.io/nikitph/laravel-runtime:1.0.0`
- One immutable application image used for `web`, `worker`, `scheduler`, and `reverb`
- PHPUnit, Pint, Larastan/PHPStan, ESLint, Prettier, and TypeScript verification
- Environment-driven configuration; no credentials or product secrets in images or Git

Do not add SSR, Kubernetes, microservices, a second backend, or Supabase-specific application
logic. Keep deployment appropriate for one DigitalOcean droplet initially while preserving a
clean path to managed PostgreSQL/Redis later.

## Authentication and account security

Retain and verify the official starter's registration, login, logout, password reset, email
verification, password confirmation, two-factor authentication, passkeys, profile editing,
and account deletion. Do not rewrite these flows merely for stylistic consistency.

New registration must create the user's first organization and owner membership atomically.

## Organizations, tenancy, roles, and permissions

Implement shared-database organization tenancy with explicit memberships.

- A user may belong to multiple organizations and has a current organization.
- Provide organization creation and switching.
- Seed four roles: `owner`, `admin`, `member`, and `viewer`.
- Permissions are explicit and centrally authorized through policies or gates.
- Owner-only destructive organization operations must not be granted through UI hiding alone.
- Every tenant-owned model has an `organization_id` foreign key and a central, default-deny
  tenant scope or equivalent query boundary.
- Tenant ownership is stamped automatically on creation where appropriate.
- Unauthenticated or missing-tenant execution must not accidentally see tenant records.
- Queue jobs, commands, scheduled tasks, and broadcasts must establish tenant context explicitly.
- Private broadcast channel authorization must enforce the same organization boundary.

This is application-enforced row isolation, not PostgreSQL's native `CREATE POLICY` RLS. Name
it accurately in code and documentation.

Include a minimal `Project` resource only to prove the full tenant boundary and UI conventions.
Keep it obviously replaceable by the first real business aggregate.

## Audit, notifications, queues, and realtime

- Record immutable audit events for important membership, organization, and project mutations.
- Provide a queued organization invitation notification and an acceptance flow.
- Configure database/Redis-compatible queues, retry behavior, `failed_jobs`, and sensible job
  timeouts without embedding infrastructure credentials.
- Configure Reverb and private-channel authorization.
- Include one small realtime delivery test or conformance endpoint.
- Preserve support for SSE/streaming responses under FrankenPHP, including explicit flushing and
  client disconnect handling in a conformance route or test fixture.
- Define scheduler behavior and ensure it runs in its own container role.

## UI productization

Create a polished, neutral SaaS shell while retaining the upstream auth implementation.

- Responsive sidebar application layout with product mark, primary navigation, organization
  switcher, current-user menu, and settings access
- Dashboard with clear hierarchy, useful status cards, recent activity, and a strong empty state
- Projects and Members screens that demonstrate list, form, authorization, feedback, and empty
  states end to end
- Landing page that presents the product foundation cleanly without claiming nonexistent features
- Consistent light and dark themes using semantic design tokens
- Accessible keyboard/focus behavior and responsive mobile navigation
- Motion used sparingly for page entrance, hierarchy, and state changes; never as decoration that
  slows routine interaction
- Toast/flash feedback for successful and failed mutations

The result should feel intentionally designed rather than like the untouched Laravel starter.
Keep reusable primitives in components and product-specific composition in pages. Avoid a large
custom component system when shadcn/ui primitives already solve the need.

## Agent-ready development conventions

Add:

- Root `AGENTS.md` with architecture boundaries, tenant-safety rules, standard commands, and
  instructions to verify changes
- `docs/START-HERE.md` for an agent beginning the first business feature
- `docs/ARCHITECTURE.md` describing request flow, tenancy, authorization, queues, realtime, and

Generated Wayfinder actions/routes are build artifacts. Regenerate them from PHP routes and
controllers rather than hand-editing generated files. The Wayfinder Vite plugin shells out to
`php artisan wayfinder:generate --with-form` during `vite build`; account for this explicitly in
the Docker build rather than assuming frontend assets can be built in a Node-only stage.

## Production image and local orchestration

Implement a multi-stage application Dockerfile:

- Composer stage installs optimized production dependencies without development packages.
- A disposable asset-builder stage contains **both PHP and Node**, plus the application files and
  Composer dependencies required for Artisan to boot. Base it on the company runtime and install
  Node only in that stage, or use another explicitly pinned PHP+Node builder.
- The asset-builder runs the production Vite build with Wayfinder generation enabled. Do not copy
  stale, host-generated Wayfinder output into the image as a workaround.
- Final stage is `FROM ghcr.io/nikitph/laravel-runtime:1.0.0`.
- Final image contains application code, production vendor dependencies, and built assets only.
- It contains no `.env`, credentials, Node runtime, `node_modules`, or development dependencies.
- It runs as the non-root user inherited from the runtime.

Provide Docker Compose services for PostgreSQL, Redis, web, worker, scheduler, and Reverb. The
four Laravel roles must use the same application image and differ only by command/configuration.
PostgreSQL and Redis need not be exposed publicly.

Provide Kamal configuration suitable for deploying the same image to an initial DigitalOcean
droplet. Keep secrets in documented environment/secrets inputs, not committed configuration.

## Verification workflow

Create a single `composer verify` entry point that performs, at minimum:

1. ESLint check
2. Prettier check
3. TypeScript type check
4. Pint check
5. Larastan/PHPStan analysis
6. PHPUnit tests
7. Production frontend build

Add CI that invokes the same contract. Test tenant isolation negatively: cross-organization
reads, writes, membership actions, and private-channel access must fail. Test missing tenant
context. Test registration provisioning, role authorization, invitation acceptance, and the
replaceable example resource.

Finish by running the full verification command and building the production Docker image from a
clean context/cache. This must prove that Wayfinder can invoke PHP during the asset build and that
the final image contains no Node runtime. Report:

- What was implemented
- Exact verification results
- Any requirement deliberately deferred and why
- The first files an agent should change when starting the business use case
